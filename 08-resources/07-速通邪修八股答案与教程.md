## 一、Activity 与生命周期

### Q1. 活动的生命周期

**答案骨架**：七状态——`onCreate`（创建，初始化布局/数据）→ `onStart`（可见但不可交互）→ `onResume`（可见可交互）→ `onPause`（失去焦点，部分遮挡）→ `onStop`（完全不可见）→ `onDestroy`（销毁）。异常情况下还有 `onRestart`。

关键场景：
- 打开新 Activity B：A `onPause` → B `onCreate→onStart→onResume` → A `onStop`
- 弹透明 Dialog：A 只走 `onPause`，不走 `onStop`
- 横竖屏切换（默认）：`onPause→onStop→onDestroy→onCreate→onStart→onResume`，可用 `configChanges` 避免重建

**推荐教程**：[Activity 生命周期 codelab](https://developer.android.com/codelabs/basic-android-kotlin-compose-activity-lifecycle) / [Gityuan Activity 启动流程](https://gityuan.com/2016/05/18/android-activity/)

### Q2. 活动的启动模式

**答案骨架**：四种 `launchMode`：

| 模式 | 特点 | 典型场景 |
| --- | --- | --- |
| `standard` | 每次新建实例，可在同一任务栈多次 | 普通页面 |
| `singleTop` | 栈顶有则复用（走 `onNewIntent`），否则新建 | 推送跳转、搜索页 |
| `singleTask` | 任务栈中已有则清掉其上所有 Activity 并复用 | 主页、登录页 |
| `singleInstance` | 独占一个任务栈，全局单例 | 系统电话、闹钟 |

补充：任务栈（Task）由 `taskAffinity` 决定归属；`FLAG_ACTIVITY_NEW_TASK` 与 `singleTask` 配合可跨应用启动。

**推荐教程**：[任务与返回栈](https://developer.android.com/guide/components/activities/tasks-and-back-stack)

## 二、Handler 与消息机制

### Q3. 线程间消息传递，消息处理

**答案骨架**：四大角色——`Handler`（发送/处理）、`Message`（载体）、`MessageQueue`（单链表实现的队列）、`Looper`（循环取消息）。

流程：子线程 `handler.sendMessage()` → 进入目标 `MessageQueue` → `Looper.loop()` 在主线程不断取 → 分发回 `Handler.handleMessage()` 在主线程执行。

关键：`Looper` 通过 `ThreadLocal` 与线程绑定；主线程 `Looper` 在 `ActivityThread.main()` 中已 `prepare` 好。

**推荐教程**：[Gityuan Handler 消息机制](https://gityuan.com/2015/12/26/handler-message-framework/)

### Q4. 延迟消息

**答案骨架**：`sendMessageDelayed` 的实现是入队时计算 `when = SystemClock.uptimeMillis() + delayMillis`；`MessageQueue.next()` 取消息时若队首 `when` 还没到，用 `nativePollOnce` 阻塞到时间点（底层 `epoll_wait`，不占 CPU）。

注意：延迟精度受消息队列拥堵影响；`postDelayed` 本质也是把 `Runnable` 包装成 `Message`。

**推荐教程**：[Gityuan Handler 消息机制](https://gityuan.com/2015/12/26/handler-message-framework/)

### Q5. Handler 消息机制？内存泄漏的问题？

**答案骨架**：机制见 Q3。

内存泄漏根因：`Handler` 持有外部 `Activity` 引用，`Message` 持有 `Handler`，`MessageQueue` 持有 `Message`。若 `Activity` 销毁时还有延迟消息未处理，`Looper` 一直持有整条链 → `Activity` 无法回收。

修复：
1. 静态内部类 `Handler` + `WeakReference<Activity>`
2. `onDestroy` 中 `handler.removeCallbacksAndMessages(null)` 清空消息
3. 优先用 `View.post` 或 `LiveData`/`协程` 替代裸 `Handler`

常见泄漏场景：静态 `Context`、未取消的 `Timer`、内部类 `Runnable`、单例回调、`Handler`、`Cursor` 未关、`WebView`。

**推荐教程**：[LeakCanary 原理与定制](https://www.cnblogs.com/dblens/p/19554504) / [LeakCanary 面试考点](https://knowledge.meng-github.com/00-android/13_Testing/08_%E5%86%85%E5%AD%98%E6%B3%84%E6%BC%8F_LeakCanary.html)

## 三、UI 与布局

### Q6. 自定义 View

**答案骨架**：三种类型——继承 `View`（重写 `onDraw`）、继承 `ViewGroup`（测量+布局子 View）、继承现有控件（扩展功能）。

核心三步：
1. `onMeasure(int, int)`：根据 `MeasureSpec` 计算自身宽高，调 `setMeasuredDimension`
2. `onLayout(boolean, l, t, r, b)`：`ViewGroup` 决定子 View 位置（`child.layout`）
3. `onDraw(Canvas)`：用 `Canvas` + `Paint` 绘制

注意：`onDraw` 中不要 `new` 对象（高频调用导致 GC）；`invalidate` 触发重绘，`requestLayout` 触发重新测量布局。

**推荐教程**：[自定义绘制](https://developer.android.com/develop/ui/views/layout/custom-views/custom-drawing) / [HenCoder 自定义 View](https://hencoder.com/tag/custom-view/)

### Q7. 布局用的比较多的？

**答案骨架**：日常以 `ConstraintLayout` 为主（扁平化、减少嵌套），`LinearLayout` 用于简单横向/纵向，`FrameLayout` 用于叠加/占位，`RecyclerView` 用于列表。`RelativeLayout` 已被 `ConstraintLayout` 取代。

### Q8. 动态约束调整布局

**答案骨架**：`ConstraintSet` 可在运行时切换约束——`clone` 当前布局 → `connect`/`clear` 修改约束 → `applyTo` 生效；`MotionLayout` 是其进阶，用 `MotionScene` 定义动画过渡。

**推荐教程**：[ConstraintLayout 性能提升](https://blog.csdn.net/kkshdgxcjsd/article/details/154289307)

### Q9. 布局优化

**答案骨架**：
1. 扁平化：用 `ConstraintLayout` 替代多层嵌套的 `RelativeLayout`+`LinearLayout`
2. 复用：`<include>`、`<merge>`（消除冗余根布局）、`ViewStub`（懒加载）
3. 减少 `overdraw`：移除多余背景，开发者选项看"过度绘制"
4. 避免 `wrap_content` 嵌套（measure 代价高）

**推荐教程**：[如何优化 Android 布局性能](https://cloud.tencent.com/developer/article/2496091)

### Q10. CoordinatorLayout 协调者布局

**答案骨架**：`CoordinatorLayout` 通过 `Behavior` 机制让子 View 监听其他子 View 的依赖/事件（滚动、位置）。典型用法：`AppBarLayout` 收起、`FloatingActionButton` 滚动隐藏、`BottomSheet`。

`Behavior` 两个核心回调：`layoutDependsOn`（是否依赖某 View）、`onDependentViewChanged`（依赖变化时响应）。

**推荐教程**：[CoordinatorLayout 基本使用](https://juejin.cn/post/7548278406836879401)

## 四、Kotlin

### Q11. Kotlin 高阶函数

**答案骨架**：参数或返回值是函数类型（`(T) -> R`）的函数。例：`let`、`map`、`forEach`。

底层：Lambda 在 JVM 上会被编译成 `FunctionN` 对象，有额外对象分配开销；用 `inline` 修饰可消除该开销（编译期把函数体展开到调用处）。

挂起函数（`suspend`）：由协程框架转为状态机，编译后带 `Continuation` 参数，可在挂起点保存/恢复栈帧。

**推荐教程**：[inline/reified 隐藏开销](https://juejin.cn/post/7640789716057554953) / [inline/crossinline/reified 的坑](https://blog.csdn.net/vitaviva/article/details/149665290)

### Q12. let/run/with/apply/also 区别

**答案骨架**：

| 函数 | 上下文引用 | 返回值 | 典型用途 |
| --- | --- | --- | --- |
| `let` | `it` | Lambda 结果 | 空判断 `?.let {}`、转换 |
| `run` | `this` | Lambda 结果 | 初始化并计算 |
| `with` | `this` | Lambda 结果 | 对同一对象多次调用 |
| `apply` | `this` | 对象本身 | 链式初始化 |
| `also` | `it` | 对象本身 | 副作用（日志）且不中断链 |

记忆：`apply`/`also` 返回对象（用于链式）；`let`/`run`/`with` 返回结果；`this` 用于"配置"，`it` 用于"使用"。

**推荐教程**：[Kotlin 作用域函数对比](https://juejin.cn/post/7504846684335947788)

### Q13. Jetpack 组件库

**答案骨架**：能说上来即可——
- `Room`：SQLite ORM，注解生成实现
- `ViewModel`：配置变化时保留 UI 数据
- `LiveData`：生命周期感知的可观察数据
- `Navigation`：单 Activity 多 Fragment 路由
- `Hilt`：Dagger 封装的依赖注入
- `WorkManager`：可延迟/可靠的后台任务
- `DataStore`：替代 `SharedPreferences`

## 五、数据库与协程

### Q14. Room 数据库

**答案骨架**：三层——`@Entity`（表）、`@Dao`（SQL 方法）、`@Database`（数据库持有者）。

关键注解：`@PrimaryKey`、`@ColumnInfo`、`@Embedded`、`@Relation`。
返回 `LiveData`/`Flow` 可观察变化；`@Transaction` 保证多表操作原子性；编译期校验 SQL。

**推荐教程**：[Room codelab](https://developer.android.com/codelabs/basic-android-kotlin-compose-persisting-data-room)

### Q15. 协程

**答案骨架**：轻量级线程，由编译器 + 调度器实现，不直接映射 OS 线程。核心：
- `suspend` 函数可在挂起点暂停不阻塞线程
- 结构化并发：父协程取消则子协程一起取消
- `CoroutineScope` 管理生命周期，避免泄漏

**推荐教程**：[协程 Playground codelab](https://developer.android.com/codelabs/basic-android-kotlin-compose-coroutines-kotlin-playground) / [Android 协程指南](https://developer.android.com/kotlin/coroutines)

### Q16. 协程线程

**答案骨架**：`Dispatchers` 决定协程在哪个线程池执行——
- `Main`：主线程（UI）
- `IO`：IO 线程池（网络/磁盘，默认 64 线程）
- `Default`：CPU 密集（线程数 = CPU 核数）
- `Unconfined`：不限定，调用者线程

切换用 `withContext(Dispatchers.IO) { }`。

### Q17. 协程作用域

**答案骨架**：
- `GlobalScope`：全局，生命周期长，易泄漏，不推荐
- `viewModelScope`：绑定 `ViewModel`，`onCleared` 自动取消
- `lifecycleScope`：绑定 `Lifecycle`，销毁自动取消
- 自建 `CoroutineScope(SupervisorJob() + Dispatchers.Main)`

`SupervisorJob`：子协程失败不影响兄弟协程；普通 `Job` 则会传播取消。

### Q18. 挂起函数

**答案骨架**：`suspend` 修饰的函数，只能在协程或其他挂起函数中调用。编译后转为 CPS（Continuation Passing Style），多一个 `Continuation` 参数，内部用状态机记录挂起点。挂起时不阻塞线程，线程可去执行别的协程。

## 六、线程与并发

### Q19. 线程启动、停止、run 方法会不会自动结束

**答案骨架**：
- 启动：`Thread.start()` 创建新线程并调 `run()`；直接调 `run()` 只是普通方法调用
- `run()` 跑完会自动结束线程
- 停止：`stop()` 已废弃（强停可能致状态不一致）；推荐用 `interrupt()` + 在循环中检查 `Thread.interrupted()` 或 `InterruptedException`

### Q20. 线程池

**答案骨架**：核心参数——`corePoolSize`、`maximumPoolSize`、`keepAliveTime`、`workQueue`、`threadFactory`、`RejectedExecutionHandler`。

提交流程：核心线程满 → 入队列 → 队列满 → 到最大线程数 → 拒绝策略（`AbortPolicy` 抛异常、`CallerRunsPolicy` 调用者执行、`DiscardPolicy` 丢弃、`DiscardOldestPolicy` 丢最旧）。

常见池：`FixedThreadPool`（固定）、`CachedThreadPool`（可缓存）、`SingleThreadExecutor`（单线程）、`ScheduledThreadPool`（定时）。

### Q21. 多线程怎么保证数据安全？

**答案骨架**：
- 互斥锁：`synchronized`（JVM 层，自动释放）、`ReentrantLock`（API 层，可中断/超时/公平）
- volatile：保证可见性 + 禁止指令重排，但不保证原子性
- 原子类：`AtomicInteger` 等（CAS）
- 并发容器：`ConcurrentHashMap`、`CopyOnWriteArrayList`
- 不可变对象：`final`、`String`
- ThreadLocal：线程隔离

## 七、网络

### Q22. 常见的 HTTP 的状态码

**答案骨架**：

| 类别 | 举例 | 含义 |
| --- | --- | --- |
| 1xx 信息 | 100 Continue | 继续发送请求体 |
| 2xx 成功 | 200 OK / 204 No Content / 206 Partial Content | 成功/无内容/范围请求 |
| 3xx 重定向 | 301 永久 / 302 临时 / 304 Not Modified | 跳转/缓存命中 |
| 4xx 客户端错 | 400 / 401 未认证 / 403 无权限 / 404 / 408 超时 / 429 限流 | 请求问题 |
| 5xx 服务端错 | 500 / 502 网关 / 503 不可用 / 504 网关超时 | 服务端问题 |

**推荐教程**：[小林 HTTP 面试题](https://www.xiaolincoding.com/network/2_http/http_interview.html)

### Q23. HTTPS 加密流程

**答案骨架**：TLS 四次握手（RSA 密钥交换为例）：

1. **ClientHello**：客户端发 TLS 版本、支持的加密套件、客户端随机数
2. **ServerHello**：服务端选定的套件、服务端随机数、**数字证书**（含公钥）
3. 客户端验证证书（CA 链 → 信任根）→ 生成 `pre-master secret`，用服务端公钥加密发回
4. 双方用两个随机数 + `pre-master` 派生 `session key`（对称密钥），后续通信用对称加密

要点：握手用非对称（安全交换密钥），通信用对称（高效）；服务端身份靠 CA 证书链保证。

**推荐教程**：[SSL/TLS 完整握手图解](https://blog.csdn.net/qq_41840843/article/details/159994308)

### Q24. TCP 三次握手

**答案骨架**：
1. 客户端发 `SYN=1, seq=x` → `SYN_SENT`
2. 服务端回 `SYN=1, ACK=1, seq=y, ack=x+1` → `SYN_RCVD`
3. 客户端发 `ACK=1, seq=x+1, ack=y+1` → `ESTABLISHED`

为什么三次：确认双方收发能力都正常；两次握手无法防止历史 `SYN` 建立无效连接。

**推荐教程**：[小林 TCP 面试题](https://xiaolincoding.com/network/3_tcp/tcp_interview.html)

## 八、数据结构与算法

### Q25. 数组和链表区别

**答案骨架**：

| 维度 | 数组 | 链表 |
| --- | --- | --- |
| 内存 | 连续 | 不连续 |
| 随机访问 | O(1) | O(n) |
| 插入删除 | O(n) 需移动 | O(1) 改指针 |
| 空间 | 无额外开销 | 每节点多一个指针 |

### Q26. 链表，怎么判断它是否有环

**答案骨架**：快慢指针——慢指针一步、快指针两步，若有环必相遇（快追上慢）；无环则快指针到 `null`。时间 O(n)，空间 O(1)。

进阶：找环入口——相遇后让一个指针回到头，两指针同速走，再次相遇点即入口。

### Q27. 栈实现队列的效果

**答案骨架**：两个栈 `in` 和 `out`——
- 入队：压入 `in`
- 出队：若 `out` 空，把 `in` 全部弹出压入 `out`，再从 `out` 弹出；否则直接从 `out` 弹出

均摊 O(1)。

### Q28. 二叉树的前序、中序、后序遍历的顺序

**答案骨架**：
- 前序：根 → 左 → 右（`DLR`）
- 中序：左 → 根 → 右（`LDR`）；BST 中序得升序
- 后序：左 → 右 → 根（`LRD`）
- 层序：按层从上到下、从左到右（用队列）

递归实现简单；非递归用栈（前/中/后）或队列（层序）。

**推荐教程**：[二叉树遍历总结](https://zhuanlan.zhihu.com/p/345762224)

### Q29. 二叉树的深度遍历和广度遍历

**答案骨架**：
- DFS（深度优先）：前/中/序遍历，递归或栈实现
- BFS（广度优先）：层序遍历，队列实现

**推荐教程**：[栈与队列实战](https://www.cnblogs.com/yangykaifa/p/19746831)

## 九、Service 与广播

### Q30. 服务的启动方式

**答案骨架**：两种——
- `startService`：启动后与调用者无生命周期绑定，需 `stopSelf` 或 `stopService` 停止；`onCreate → onStartCommand`（可多次）→ `onDestroy`
- `bindService`：与绑定者绑定，所有绑定者解绑后销毁；`onCreate → onBind → onUnbind → onDestroy`

混合启动：先 `start` 再 `bind`，需 `stop` + `unbind` 都执行才销毁。

**推荐教程**：[Service 面试题干货](https://developer.aliyun.com/article/928972) / [Service 生命周期](https://cloud.tencent.com/developer/article/2531026)

### Q31. Activity 和 Service 有什么差异点、共同点？

**答案骨架**：
- 共同：都是四大组件，都在主线程，都需在 `AndroidManifest` 注册
- 差异：Activity 有 UI、可与用户交互、有任务栈概念；Service 无 UI、后台运行、不与用户直接交互；生命周期不同

### Q32. 广播的注册方式

**答案骨架**：
- 静态注册：在 `AndroidManifest` 的 `<receiver>` 声明，应用未启动也能接收（Android 8.0 后大幅限制，仅部分系统广播生效）
- 动态注册：代码 `registerReceiver`，跟随注册者生命周期，需 `unregisterReceiver` 防泄漏；优先级 `intent-filter` 的 `priority`

广播类型：标准广播（异步，无序）、有序广播（按优先级传递，可截断）、本地广播（`LocalBroadcastManager`，应用内）。

**推荐教程**：[BroadcastReceiver 面试题](https://cloud.tencent.com/developer/article/2531027)

### Q33. 一个广播接收到了一个信息之后，要做一个耗时的操作，怎么处理

**答案骨架**：`BroadcastReceiver` 的 `onReceive` 默认 10 秒超时（前台），超时 ANR。

正确做法：
- `onReceive` 内启动 `Service`（`startService` 或 `JobIntentService`/`WorkManager`）处理耗时
- 或用 `goAsync()` 拿 `PendingResult`，在子线程处理后调 `finish()`（延长到约 30 秒）
- 不要直接 `new Thread`（系统可能在 `onReceive` 返回后杀进程）

- **建造者**：`AlertDialog.Builder`、`Request.Builder`、`RoomDatabase.Builder`

**推荐教程**：[Java 23 种设计模式全归纳](https://cloud.tencent.com/developer/article/1602270) / [一文速通 23 种设计模式](http://blog.mvui.cn/index.php/detail/106170.html)

### JSON 解析（邪修第 9 题）

**答案骨架**：Android 常用 `Gson`（反射，简单慢）和 `Moshi`（Kotlin 友好，`KotlinJsonAdapterFactory`，codegen 编译期生成，快且安全）。`data class` + `@Json` 注解处理字段名映射；注意非空字段缺失会崩。

**推荐教程**：[Gson 与 Moshi 对比](https://blog.csdn.net/2600_94959881/article/details/157318412) / [从 Gson 迁移到 Moshi](https://icode.best/i/758747421214361)

### AI 概念（邪修末尾提到）

- **Agent**：能感知环境 + 决策 + 调用工具 + 循环执行的 AI 系统；核心是 Agent Loop（感知→思考→行动→观察）
- **MCP**：Model Context Protocol，Anthropic 提出的模型与外部工具/资源通信标准；三个 primitives：`tools`、`resources`、`prompts`
- **Skill**：可复用的领域知识包（本文档就是一种 skill）
- **RAG**：检索增强生成，先从向量库检索相关片段，再拼进 prompt 给 LLM 生成；解决幻觉 + 私有数据

**推荐教程**：[OpenAI 实用 Agent 指南](https://openai.com/business/guides-and-resources/a-practical-guide-to-building-ai-agents/) / [MCP 架构](https://modelcontextprotocol.io/docs/learn/architecture)
